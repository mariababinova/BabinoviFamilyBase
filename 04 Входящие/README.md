# Входящие

Сюда попадают новые медицинские файлы до переноса в основную базу.

## Поток обработки

1. `00 Новые файлы` — сюда кладутся свежие PDF, фото и текстовые заметки.
2. `10 Черновики AI` — агент складывает первичный разбор.
3. `20 На проверке` — черновики, которые нужно проверить человеку.
4. `30 Одобрено` — пользователь подтвердил разбор, можно переносить в основную базу.
5. `90 Обработано` — исходники и черновики после успешного переноса.
6. `99 Ошибки` — файлы, которые не удалось надежно распознать.

## Локальный агент

Команды запускаются из папки `06 Сайт`.

```powershell
npm run agent:intake
npm run agent:intake -- --dry-run
npm run agent:promote
npm run agent:promote -- --dry-run
```

## Обычный запуск через общий конвейер

Для повседневной работы лучше запускать не отдельные агенты, а общий конвейер из папки сайта:

```powershell
cd <resolved 06 *>; npm run update:base
```

Где `<resolved 06 *>` — фактическая папка `06 Сайт` в этом репозитории.

Что делает общий запуск:

1. Проверяет новые файлы в `04 Входящие/00 Новые файлы`.
2. Создает черновики AI в `04 Входящие/10 Черновики AI`.
3. Переносит только одобренные черновики со `status: approved`.
4. Обновляет кандидаты показателей и задач.
5. Пересобирает производные данные сайта.
6. Записывает отчет запуска в `09 Наблюдение/update-base-state.json`.

Безопасная проверка без записи:

```powershell
cd <resolved 06 *>; npm run update:base -- --dry-run
```

Запуск без AI, когда нужно проверить только локальные шаги:

```powershell
cd <resolved 06 *>; npm run update:base -- --no-ai
```

Важно: `--no-ai` только отключает обращения к OpenAI. Это не dry-run: локальные шаги, у которых есть что записывать, могут менять файлы. Для проверки без записи используйте `--dry-run`.

Что открыть после запуска:

- `04 Входящие/10 Черновики AI` — новые черновики документов.
- `04 Входящие/20 На проверке` — документы, которые ждут ручного решения.
- `07 Показатели/metric_candidates.json` — новые показатели и референсы перед попаданием в `metrics.json`.
- `08 Задачи/task_candidates.json` или локальную страницу `/task-review` — новые задачи перед подтверждением.
- `09 Наблюдение/update-base-state.json` — технический отчет: какие шаги прошли, что пропущено, где ошибка.

AI-черновики, новые показатели и задачи не становятся финальными медицинскими данными без ручного подтверждения.

## Ключи и инструменты

Проверить окружение можно из папки `06 Сайт`:

```powershell
node scripts/doctor-env.mjs
```

Для полноценного чтения PDF и фото нужен `OPENAI_API_KEY` в `.env` корня репозитория или в переменных окружения. `OPENAI_MODEL` можно не менять, если нет отдельной причины.

Для извлечения текста из PDF и поиска лабораторных референсов нужна Python-библиотека PyMuPDF:

```powershell
py -m pip install PyMuPDF
```

Если `py` недоступен:

```powershell
python -m pip install PyMuPDF
```

Секреты нельзя коммитить. Production-переменные вроде `MEDS_GITHUB_TOKEN`, `SITE_AUTH_PASSWORD` и ключей документов нужны только для серверных review API, приватного хостинга или публикации документов.

`agent:intake` просматривает `00 Новые файлы`, создает черновик в `10 Черновики AI` и запоминает обработанные файлы в `intake-state.json`, чтобы не плодить дубликаты.

Сейчас `agent:intake` не просто смотрит на имя файла: для PDF и фото он отправляет содержимое в OpenAI Responses API, извлекает дату, человека, тип документа, врача/клинику, показатели и задачи, затем переименовывает файл в `00 Новые файлы` в канонический формат вроде `2026-05-02 Артём — Кардиолог.pdf`. Для этого нужен `OPENAI_API_KEY` в `.env` или переменных окружения.

Если чтение не удалось, файл остается в `00 Новые файлы`, чтобы его можно было повторно обработать после исправления ключа или промпта. Переносить ошибки в `99 Ошибки` можно явно: `npm run agent:intake -- --move-errors`.

`agent:promote` берет только черновики со статусом `approved`, создает запись `medical_event` в папке нужного члена семьи и копирует туда исходные файлы. Если в черновике нет уверенно указанного человека или даты, событие не создается.

## Правило безопасности

Агент не должен писать финальные медицинские события напрямую в `01 Члены семьи` без явного подтверждения. Сначала создается черновик со статусом `needs_review`.
