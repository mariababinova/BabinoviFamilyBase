# Памятка проекта для Codex

## Задачи и рекомендации
- Новые рекомендации из документов сначала попадают в `08 Задачи/task_candidates.json` со статусом `needs_review`.
- В `08 Задачи/tasks.json` попадают только подтверждённые задачи: уже существующие задачи или кандидаты со статусом `approved`.
- Отклонённые рекомендации остаются в `task_candidates.json` со статусом `rejected` и не показываются как задачи.
- Временный служебный экран проверки кандидатов на сайте: `/task-review`; позже его можно заменить кнопками подтверждения прямо в интерфейсе.
- Для локальной проверки кнопок `/task-review` без Vercel API можно запускать из `06 Сайт`: `node scripts/local-task-review-api.mjs`; он поднимает fallback API на `127.0.0.1:4330` и пишет в локальные JSON.
- В продакшене подтверждение/отклонение рекомендаций идёт через `/api/tasks/review`; для записи в GitHub нужен `MEDS_GITHUB_TOKEN` с доступом на запись к репозиторию.
- Тестовый кейс этапа 4: документ Маши `2022-09-05 Маша — гистология и ВПЧ` породил кандидата про онкоцитологию каждые 6 месяцев; кандидат подтверждён и стал задачей Маши на `2026-09-05`.
- Рекомендация из врачебного документа не становится задачей автоматически, если это не отдельное понятное действие для семьи.
- Если несколько рекомендаций относятся к одному реальному действию, их нужно объединять в одну сборную задачу.
- Сборная задача хранится как `kind: grouped_task`, получает `task_page: true`, список `items` и все связанные `grouped_source_event_ids`.
- Карточка сборной задачи на сайте должна вести на отдельную страницу `/tasks/<slug>`, а не на одно случайное событие-источник.
- На странице сборной задачи нужно показывать, что сделать, из каких событий/документов это взято, и давать ссылки на каждое исходное медицинское событие.
- Обычные одиночные задачи могут вести прямо на связанное событие.
