# Памятка проекта для Codex

Эта заметка нужна, чтобы можно было начать новый чат с Codex и быстро восстановить весь контекст по семейной медицинской базе.

В начале нового чата полезно написать: `прочитай 00 Главная/Памятка проекта для Codex.md и продолжай по ней`.

## Суть проекта

Мы делаем семейную медицинскую базу знаний.

Цель: не просто хранить PDF и фотографии анализов, а превращать их в понятную медицинскую историю семьи:

- у каждого человека есть свой профиль;
- документы привязаны к датам, событиям, врачам и разделам;
- анализы собираются в показатели и динамику;
- рекомендации врачей превращаются в задачи, но не автоматически бездумно;
- спорные извлечения уходят на проверку;
- сайт нужен как спокойный интерфейс для чтения базы, а не как медицинская система постановки диагнозов.

Важный принцип: база помогает ничего не потерять и подготовиться к врачу, но не ставит диагнозы и не заменяет врача.

## Как устроены медицинские события

Медицинское событие - это смысловая запись: приём, анализ, обследование, процедура, выписка и т.д.

Обычно событие лежит внутри папки человека и специальности, например:

`01 Члены семьи/Маша/16 Эндокринолог/16 Эндокринолог 2026/2026-05-23 Маша — эндокринолог.md`

Важные поля frontmatter:

- `id`
- `type: medical_event`
- `person`
- `date`
- `event_type`
- `specialty`
- `doctor`
- `clinic`
- `status`
- `importance`
- `follow_up_date`
- `source_files`
- `tags`

Если врач/клиника неизвестны, лучше оставить пусто, чем выдумывать.

## Задачи

Файлы:

- `08 Задачи/tasks.json` - подтверждённые задачи.
- `08 Задачи/task_candidates.json` - кандидаты на задачи.

Правила:

- Новые рекомендации из документов сначала попадают в `task_candidates.json` со статусом `needs_review`.
- В `tasks.json` попадают только подтверждённые задачи или явно добавленные вручную важные задачи.
- Отклонённые рекомендации остаются в `task_candidates.json` со статусом `rejected` и не показываются как реальные задачи.
- Рекомендация из врачебного документа не становится задачей автоматически, если это не отдельное понятное действие для семьи.
- Если несколько рекомендаций относятся к одному реальному действию, их нужно объединять в одну сборную задачу.

Сборная задача:

- `kind: grouped_task`
- `task_page: true`
- `items: [...]`
- ссылки на исходные события/документы
- страница `/tasks/<slug>`

Одиночная задача может вести прямо на событие.

Временный экран проверки задач:

- `/task-review`

Локальный fallback API для проверки:

```bash
cd "06 Сайт"
node scripts/local-task-review-api.mjs
```

Продакшен API:

- `/api/tasks/review`
- для записи в GitHub нужен `MEDS_GITHUB_TOKEN` с правом записи в репозиторий.

## Показатели и анализы

Это был большой этап 5.

Что уже сделано:

- показатели собираются в `metrics`;
- у показателя есть canonical id, например `hemoglobin`, `ferritin`, `tsh`;
- разные лабораторные названия связываются с единым показателем;
- показатели показываются на сайте списком и отдельными страницами;
- добавлены статусы по референсу, если референс известен;
- если референса нет, не надо агрессивно красить показатель, нужно писать "референс не указан";
- декоративные полоски в карточках показателей заменялись/переосмыслялись: пользователь хочет информативную цветовую логику, а не бессмысленный декор.

Важная медицинская логика:

- не делать один глобальный референс "навсегда";
- нормы зависят от лаборатории, пола, возраста, беременности/лактации, метода измерения и единиц;
- в `metrics.json` нужно хранить фактическое значение, единицы, лабораторию, исходный референс из документа;
- в справочнике показателей хранить canonical metric id;
- при импорте нормализовать единицы, если возможно;
- abnormal status брать либо из лабораторного флага, либо вычислять по референсу конкретного документа;
- если референса нет, писать "референс не указан".

## Референсы

Референсы берём не из глобального справочника, а прежде всего из конкретного лабораторного документа.

Нормальная модель:

- документ содержит исходный референс;
- агент извлекает значение, единицы, референс, лабораторию, дату;
- сайт показывает статус относительно референса именно этого документа;
- если в документе референса нет или он не извлечён, статус неизвестный.

Будущая доработка:

- улучшить извлечение из PDF/PNG так, чтобы агент забирал соседнюю колонку референса, единицы, лабораторию, метод/материал;
- сделать отдельный `lab-analysis-agent`;
- складывать черновик анализа сначала в `04 Входящие/20 На проверке` или `09 Наблюдение/lab-analysis.json`, а не сразу в сайт.

## Автоматизация обработки новых документов

Желаемый принцип:

Если добавляется новый документ, из этого должен запускаться общий процесс:

1. распознать/разобрать документ;
2. понять человека, дату, тип документа, врача/клинику/специальность;
3. создать или обновить медицинское событие;
4. привязать PDF/фото к событию;
5. извлечь показатели, если это анализы;
6. извлечь задачи-кандидаты, если есть рекомендации;
7. обновить сводки врачу;

## Дизайн-логика

Пользователь не хочет декоративный шум вместо смысла.

Особенно по показателям:

- цвет должен означать статус;
- зелёный - в референсе;
- красный/жёлтый - ниже или выше нормы, но всегда относительно референса;
- серый/нейтральный - нет референса или неизвестно;
- не надо оставлять голубые/розовые/красные полоски, если они ничего не значат;
- не надо дублировать одно и то же бессмысленными бейджами и полосками.

Для медицинского интерфейса важны:

- спокойная читаемость;
- плотная, но понятная структура;
- минимум декора, который мешает;
- цвет только там, где он несёт смысл.

## Публикация

Обычный publish workflow:

```bash
git status --short --branch
git pull --ff-only origin main
cd "06 Сайт"
npm run validate:ops
npm run validate:data
npm run check
npm run build
cd ..
git status --short
git diff --stat
git add -- <только нужные файлы>
git commit -m "<сообщение>"
git push origin main
gh run list --limit 5
gh run watch <run-id>
```

Важно:

- не коммитить `.env` и секреты;
- не коммитить случайные временные файлы;
- не коммитить сырые входящие файлы из `04 Входящие/00 Новые файлы`, если пользователь явно не просил их опубликовать;
- перед `git pull` смотреть статус, чтобы не потерять локальные изменения;
- не использовать `git reset --hard` и не откатывать чужие изменения.

## Как работать с параллельными чатами

Проблема уже возникала: два чата работали с одним проектом параллельно, из-за чего версии сайта и ссылки расходились.

Чтобы не путаться:

- лучше вести активную разработку в одном основном чате;
- второй чат можно использовать только для загрузки/описания новых файлов, но он должен в конце явно сделать commit/push или передать список изменений;
- перед началом любого чата нужно проверить `git status`, `git pull --ff-only origin main`, последние коммиты;
- если другой чат уже что-то запушил, сначала подтянуть `main`;
- если работа идёт параллельно, нужно разделять задачи: один чат только intake, другой только сайт, но обязательно синхронизироваться через git.

Лучший шаблон сообщения для нового чата:

```text
Прочитай 00 Главная/Памятка проекта для Codex.md.
Рабочий репозиторий: C:\Users\Professional\Documents\Бабиновы\Медицинская база знаний\_github_MedsDataBase.
Перед изменениями сделай git status и git pull --ff-only origin main.
Ничего не публикуй без проверки validate/check/build.
```
