# План реализации статического медицинского дашборда

## 1. Цель

Сделать простой, быстрый и понятный веб-дашборд поверх Obsidian-хранилища `MedsDataBase`, чтобы человек без опыта работы с Obsidian мог удобно смотреть семейную медицинскую базу: профили, события, документы, ближайшие задачи, таймлайны и сводки.

Сайт не должен изменять данные. Источник правды остаётся в приватном git-репозитории и локальном Obsidian-хранилище. Любые изменения вносятся через Obsidian/файлы, затем коммитятся и пушатся в GitHub. После `push` GitHub Actions пересобирает статический сайт и публикует его на GitHub Pages.

Публичная сборка GitHub Pages показывает структурированные заметки, статусы, индексы и метаданные документов, но не публикует оригиналы PDF/JPG/PNG. Оригиналы остаются в приватном репозитории и могут попадать в `dist` только в явном локальном/приватном режиме.

## 2. Не цели

- Не делать редактирование заметок через сайт.
- Не делать загрузку PDF/фото через сайт.
- Не делать медицинские рекомендации, диагнозы или интерпретации сверх уже внесённых заметок.
- Не делать полноценную авторизацию на первом этапе.
- Не заменять Obsidian как рабочее место для ведения базы.

## 3. Базовая схема

```text
Obsidian локально
  -> Markdown, YAML, PDF, JPEG в приватном репозитории
  -> git commit / git push
  -> GitHub Actions
  -> build статического сайта без raw-документов
  -> GitHub Pages
  -> публичный read-only дашборд с метаданными и заметками
```

## 4. Технологический выбор

Рекомендуемый стек: Astro + TypeScript.

Причины:

- Astro хорошо подходит для статических сайтов без backend.
- Можно читать Markdown и YAML на этапе сборки.
- Можно собрать GitHub Pages без серверной части.
- Получится быстрый сайт с минимальным JavaScript.
- Позже можно добавить клиентский поиск, фильтры и графики без смены архитектуры.

Альтернатива: Next.js static export. Его имеет смысл брать, если заранее планируется сложное клиентское состояние и много интерактивных компонентов. Для текущего read-only дашборда Astro проще и спокойнее.

## 6. Источник данных

Сайт должен читать данные не из ручного JSON, а из текущей структуры хранилища. При этом важно строго разделять первичные данные, служебные шаблоны и сгенерированные индексы.

```text
01 Члены семьи/   основной источник профилей, событий и документов
04 Входящие/      необработанные входящие файлы, отдельная служебная зона
03 Шаблоны/       не источник данных сайта
05 Индексы/       не источник данных сайта, только Obsidian-индексы
06 Сайт/          код сайта, не источник медицинских данных
Архив/            не публиковать в MVP, если явно не включено
```

Основной источник событий: markdown-файлы с `type: medical_event`, найденные только внутри `01 Члены семьи/**/*.md`.

Основной источник профилей: markdown-файлы с `type: person_profile`, найденные только внутри `01 Члены семьи/**/*.md`.

Дополнительный источник документов: все `.pdf`, `.jpg`, `.jpeg`, `.png`, найденные внутри `01 Члены семьи`, даже если у них пока нет markdown-события.

`03 Шаблоны` нужно исключать из парсинга всегда. В идеале шаблоны стоит перевести на `type: template`, чтобы они даже теоретически не выглядели как реальные события.

`05 Индексы` также исключаются всегда: это производный Obsidian-артефакт, а не вход сайта. Дашборд строит собственные индексы из первичных заметок.

`04 Входящие` не смешивается с медицинским таймлайном. Файлы оттуда можно показывать только в отдельной служебной зоне “Неразобранное”, если включён флаг `publishInbox: true`. По умолчанию они не попадают в карточки людей, задачи и медицинские события.

## 7. Нормализованная модель данных

### Person

```ts
type Person = {
  id: string;
  slug: string;
  name: string;
  profilePath: string;
  birthDate?: string;
  bloodType?: string;
  rhFactor?: string;
  allergies?: string[];
  chronicConditions?: string[];
  currentMedications?: string[];
  currentTreatments?: string[];
  importantStatus?: string;
  eventCount: number;
  documentCount: number;
  nextTasks: Task[];
  dataWarnings: DataIssue[];
};
```

Поля профиля сейчас в основном находятся в теле markdown-заметки, а не в YAML. Поэтому для `Person` нужен парсер секций `Основная информация`, `Важное`, `Последние важные события`, `Ближайшие задачи`. Даты рождения лучше постепенно нормализовать в ISO `YYYY-MM-DD`, но MVP должен уметь читать текущий формат `DD.MM.YYYY`.

Возраст не должен сохраняться как статический текст в build-output. Его нужно вычислять на клиенте от текущей даты браузера, чтобы сайт не устаревал без нового деплоя.

### MedicalEvent

```ts
type MedicalEvent = {
  id: string;
  slug: string;
  title: string;
  person: string;
  personSlug: string;
  date: string;
  year: string;
  eventType: string;
  specialty: string;
  doctor?: string;
  clinic?: string;
  status: string;
  importance: "low" | "normal" | "high" | "critical";
  followUpDate?: string;
  sourceFiles: SourceDocument[];
  tags: string[];
  summary: string[];
  track: string[];
  nextActions: string[];
  markdownPath: string;
  folderPath: string;
  dataWarnings: DataIssue[];
};
```

### SourceDocument

```ts
type SourceDocument = {
  id: string;
  slug: string;
  fileName: string;
  originalPath: string;
  publicUrl?: string;
  extension: "pdf" | "jpg" | "jpeg" | "png";
  person?: string;
  eventId?: string;
  specialty?: string;
  date?: string;
  isLinkedToEvent: boolean;
  isInboxItem: boolean;
  isOriginalPublic: boolean;
};
```

### Task

```ts
type Task = {
  id: string;
  person: string;
  personSlug: string;
  dueDate?: string;
  specialty: string;
  sourceType: "event" | "profile" | "manual";
  sourceEventId?: string;
  sourceEventSlug?: string;
  sourcePath: string;
  sourceTitle: string;
  title: string;
  status: "computed_on_client";
  actionText: string;
};
```

Статус задачи нельзя жёстко запекать на дату сборки. Категории `overdue`, `this_week`, `this_month`, `later`, `unknown` вычисляются в браузере относительно текущей даты. Дополнительно можно настроить nightly GitHub Actions rebuild, но интерфейс всё равно должен корректно работать без ежедневного деплоя.

Задачи могут приходить не только из события, но и из профиля человека или ручного служебного списка. Поэтому `dueDate`, `sourceEventId` и `sourceEventSlug` не обязательны. Для задач без даты используется секция `Без даты`, а ссылка ведёт на `sourcePath`/`sourceTitle`.

### DataIssue

```ts
type DataIssue = {
  id: string;
  severity: "fatal" | "skip" | "warn" | "info";
  entityType: "event" | "person" | "document" | "task" | "build";
  entityPath?: string;
  message: string;
  suggestedFix?: string;
};
```

## 8. Парсинг Markdown

На этапе сборки `loadVault.ts` должен:

1. Найти `.md` файлы только внутри `../01 Члены семьи/**/*.md`.
2. Явно исключить `03 Шаблоны`, `05 Индексы`, `06 Сайт`, `.obsidian`, `.git`, `Архив`, `node_modules`, `dist`.
3. Прочитать YAML frontmatter.
4. Отобрать события по `type: medical_event`.
5. Отобрать профили по `type: person_profile`.
6. Из тела заметки события извлечь секции:
   - `Краткий итог`
   - `Что важно отследить`
   - `Что делать дальше`
   - `Назначения`
   - `Контроль в июле`
   - `Планово`
7. Из тела заметки профиля извлечь секции:
   - `Основная информация`
   - `Важное`
   - `Последние важные события`
   - `Ближайшие задачи`
8. Найти связанные документы по `source_files`.
9. Найти документы без события.
10. Построить единый `DashboardData`.

Важно: парсер должен быть терпимым. Если какое-то поле отсутствует, сайт не падает, а показывает пустое состояние или “не указано”.

### Разрешение `source_files`

В YAML сейчас лежат в основном голые имена файлов, поэтому алгоритм должен быть детерминированным:

1. Искать файл рядом с `.md` событием.
2. Если не найден, искать внутри той же годовой папки.
3. Если не найден, искать внутри ветки того же человека.
4. Если не найден, искать глобально внутри `01 Члены семьи`.
5. Если найдено несколько совпадений, создать `DataIssue` с severity `warn` или `fatal` в зависимости от того, можно ли выбрать ближайший файл без двусмысленности.

Глобальный матч вне ветки текущего человека нельзя привязывать автоматически. В этом случае нужно создать `DataIssue` severity `warn` и оставить документ в списке непривязанных, пока в YAML не появится явный относительный путь.

Для сравнения имён нужно нормализовать Unicode, path separators, регистр расширения и лишние пробелы. На Linux runner GitHub Actions пути case-sensitive, поэтому нельзя полагаться на поведение Windows.

### Markdown в теле заметок

Для MVP допускается plain-text extraction из нужных секций, но он должен:

- превращать Obsidian-ссылки `[[Название]]` в обычный текст или внутренние ссылки, если цель найдена;
- не показывать пользователю сырой YAML;
- не показывать необработанные `[[...]]` в карточках;
- сохранять списки как списки;
- таблицы и сложную разметку показывать в detail-view, а в карточках давать только короткий plain-text summary.

Если понадобится полноценный рендер markdown, лучше использовать `remark`/`rehype` и отдельный обработчик wikilinks.

### Slug-стратегия

URL не должны строиться из кириллических имён напрямую.

- `personSlug`: транслитерация имени + fallback hash.
- `eventSlug`: `id`, если он валиден и уникален; иначе path-based slug с hash.
- `documentSlug`: hash исходного пути + безопасная часть имени файла.

Отображаемые имена остаются русскими. Slug нужен только для URL и внутренних ссылок.

## 9. Документы и PDF

В публичной сборке GitHub Pages оригиналы медицинских документов не публикуются. Дашборд показывает карточки документов, связь с событием, человека, дату, направление и статус нормализации, но вместо кнопки открытия оригинала показывает нейтральное состояние “Оригинал скрыт”.

Причина: всё, что попадает в `dist`, становится публичным. Даже если репозиторий приватный, опубликованный Pages-артефакт нужно считать открытым для чтения.

Оригиналы можно включать только в явном приватном режиме:

```text
npm run build:with-documents
```

Этот режим нужен для локальной проверки или отдельного закрытого хостинга, но не должен использоваться для публичного GitHub Pages.

Если документы всё-таки публикуются в приватном режиме, рекомендуемый формат пути:

```text
/MedsDataBase/files/documents/{hash}-{safe-file-name}.pdf
```

Зачем нужен hash в приватном document URL:

- кириллические длинные имена безопасно работают не во всех окружениях;
- могут встретиться одинаковые имена PDF в разных папках;
- URL будет стабильнее и короче.

При этом в интерфейсе нужно показывать нормальное человеческое имя файла, а не hash.

### Build pipeline для документов

Копирование документов не должно происходить в production-сборке. Основной pipeline должен генерировать данные, собирать Astro и затем проверять, что raw PDF/JPG/PNG не попали в `dist`.

Рекомендуемый production pipeline:

```text
npm run validate:ops
npm run validate:data
npm run validate:assets
npm run build
npm run validate:pages-artifact
```

`npm run build` генерирует `src/generated/dashboard-data.json` и `src/generated/document-manifest.json` локально/в CI, но эти JSON не коммитятся. В публичной сборке у документов нет `publicUrl`, а `validate:pages-artifact` падает, если в `dist` обнаружены `.pdf`, `.jpg`, `.jpeg` или `.png`.

Для приватной сборки с оригиналами используется отдельная команда:

```text
npm run build:with-documents
```

Она явно включает `publicUrl`, собирает Astro и только затем копирует документы в `dist/files/documents`.

Для локальной разработки с открытием оригиналов нужен отдельный режим:

```text
npm run dev:data
npm run dev
```

`dev:data` генерирует manifest и копирует документы в `06 Сайт/public/files/documents`, потому что `astro dev` не отдаёт файлы из `dist`. Этот режим считается локальным и приватным.

### Контроль размера документов

Нужна команда `npm run validate:assets`.

Пороговые правила:

- warning, если суммарный размер документов больше 500 MB;
- fatal, если суммарный размер документов больше 900 MB;
- warning для одного файла больше 50 MB;
- fatal для одного файла больше 100 MB, если не выбран альтернативный hosting strategy.

Причина: GitHub Pages имеет ограничения по размеру опубликованного сайта, времени деплоя и bandwidth. Если база начнёт быстро расти, запасные варианты: Cloudflare Pages, Cloudflare R2 для документов, GitHub Releases для тяжёлых файлов или отдельный сервер для assets.

### Просмотр документа

У документа должен быть отдельный detail-route `/documents/{documentSlug}`, а не только ссылка “открыть PDF”.

Экран документа показывает:

- имя файла;
- человек, дата, направление, если определены;
- связанное событие, если есть;
- статус “привязан к записи” или “нет описания события”;
- публичный статус должен называться “привязан к записи” или “нет описания события”, а не “карточка”;
- действия: “Назад к событию”, “Назад к документам”;
- если `publicUrl` доступен в приватной сборке: “Открыть оригинал”, “Скачать”;
- если `publicUrl` отсутствует в публичной сборке: заметное состояние “Оригинал скрыт”.

Для изображений нужен встроенный просмотр с масштабированием только в приватной сборке с документами. Для PDF можно сначала использовать iframe/embed, но обязательно оставить fallback “Открыть оригинал в новой вкладке”, потому что браузеры и мобильные устройства по-разному показывают PDF.

## 10. Маршруты

### `/`

Главная панель:

1. Срочные задачи: просроченные и ближайшие.
2. Карточки членов семьи.
3. Последние события.
4. Компактное предупреждение “есть документы без описания события”, если такие есть.

Главная не должна превращаться в панель всех фильтров сразу. Глубокие фильтры живут в разделах “Все события”, “Обзор”, “Документы” и “Поиск”.

### `/events`

Общий журнал событий:

- все медицинские события в обратной хронологии;
- фильтр по человеку;
- фильтр по году;
- фильтр по направлению;
- фильтр по типу записи;
- фильтр “есть документы / нет документов”;
- переключение вида: список и компактный таймлайн.

Это базовый сценарий для пользователя, который хочет просто посмотреть “всё, что было”, не заходя в профиль конкретного человека.

### `/people/{personSlug}`

Профиль человека:

- основная информация из профиля;
- важное: аллергии, хронические заболевания, препараты, курсы;
- счётчики: события, документы, ближайшие задачи;
- таймлайн;
- фильтры по направлению и году;
- документы человека;
- список задач.

### `/events/{eventSlug}`

Карточка события:

- дата;
- человек;
- тип;
- направление;
- врач;
- клиника;
- краткий итог;
- что важно отследить;
- что делать дальше;
- назначения, если есть;
- связанные PDF/изображения;
- соседние события того же человека;
- события того же направления.

### `/tasks`

Задачи показываются не таблицей по умолчанию, а секциями действий:

- `Просрочено`;
- `На этой неделе`;
- `В этом месяце`;
- `Позже`;
- `Без даты`.

Каждая задача должна иметь понятный `actionText`, человека, направление, дату контроля, источник и быстрый переход к событию. Табличный вид можно добавить позже как альтернативный desktop-режим, но mobile-first представление должно быть карточками.

### `/documents`

Все документы:

- фильтр по человеку;
- фильтр по типу файла;
- фильтр “привязан к записи / нет описания события”;
- поиск по имени файла;
- переход к событию, если документ связан;
- переход в detail-view документа;
- в приватной сборке с `publicUrl` открыть оригинал PDF/изображения в новой вкладке.

### `/documents/{documentSlug}`

Карточка документа:

- метаданные документа;
- связанное событие;
- встроенный просмотр, если доступен `publicUrl`;
- fallback-ссылки на оригинал и скачивание.

### `/overview`

Обзор по двум независимым осям:

- направление: кардиология, терапия, ЛОР, гинекология, педиатрия и т.д.;
- тип записи: приём, консультация, анализ, обследование, оценка риска, заключение.

Нельзя смешивать “Анализы” и “Кардиология” как элементы одного списка, потому что это разные измерения. На странице `overview` должны быть два фильтра: `specialty` и `eventType`.

Для каждого выбранного среза:

- количество событий;
- последние события;
- люди, у которых есть данные;
- документы, связанные с этим срезом.

### `/search`

Клиентский поиск:

- по названию события;
- по человеку;
- по врачу;
- по клинике;
- по краткому итогу;
- по назначениям;
- по секциям `Что важно отследить` и `Что делать дальше`;
- по `actionText` задач;
- по важным полям профиля человека;
- по имени PDF;
- по тегам.

На первом этапе достаточно простого поиска по JSON-индексу без внешнего сервиса.

UX поиска:

- один глобальный поиск доступен из шапки;
- отдельная страница `/search` показывает расширенные результаты;
- результаты группируются: `События`, `Документы`, `Люди`, `Задачи`;
- совпадения подсвечиваются;
- у результата есть сниппет найденного фрагмента;
- пустое состояние предлагает сбросить фильтры или изменить запрос;
- если активны фильтры, пользователь видит чипы фильтров и кнопку “Сбросить”.

## 11. UI/UX принципы

Целевая аудитория: человек, которому не нужен Obsidian и структура файлов. Интерфейс должен быть похож на спокойный личный медицинский кабинет, а не на техническую документацию.

Принципы:

- Главный экран сразу показывает полезное: люди, задачи, последние события.
- Не показывать обычному зрителю термины Obsidian, GitHub, git, YAML, commit, push, Codex.
- Не показывать пути файлов и внутренние id без необходимости.
- Карточка каждого PDF должна открываться максимум за 1-2 клика.
- Оригинал PDF открывается только в приватной сборке, где явно включён `publicUrl`.
- События должны быть отсортированы по времени.
- Важные задачи должны быть видны без поиска.
- Фильтры должны быть очевидными и обратимыми.
- На мобильном всё должно оставаться читаемым.

Технические инструкции для владельца базы должны жить отдельно: `/owner-guide`. Эта страница не является частью основного сценария просмотра и не должна быть заметной первичной кнопкой.

## 14. Кнопки и действия

Так как сайт read-only, кнопки не должны имитировать редактирование.

Разрешённые действия:

- открыть событие;
- открыть карточку документа;
- открыть PDF/изображение только в приватной сборке с `publicUrl`;
- перейти к человеку;
- фильтровать;
- искать;
- сбросить фильтры;
- скопировать название события.

Нельзя делать первичную кнопку “Добавить запись”: она создаёт ожидание, что сайт умеет менять данные.

Вместо неё можно сделать вторичный служебный пункт “Как обновляется база” в `/owner-guide`. Там допустимы инструкции:

- где редактировать заметки;
- куда класть новые PDF;
- как обновить индексы;
- как сделать commit/push;
- что произойдёт после деплоя.

В основном интерфейсе должны быть только действия просмотра: открыть карточку, фильтровать, искать. Скачать/открыть оригинал документа можно показывать только в приватной сборке с `publicUrl`.

## 15. Цвета и стиль

Тон: спокойный, рабочий, медицинский, но не больнично-холодный.

Предлагаемая палитра:

- фон: почти белый `#f7f8fa`;
- основной текст: графитовый `#1f2933`;
- вторичный текст: серый `#64748b`;
- границы: `#d7dde5`;
- акцент действий: сине-зелёный `#0f766e`;
- задачи/предупреждения: янтарный `#b7791f`;
- просроченное: красный `#b42318`;
- документы: спокойный синий `#2563eb`;

Важно не делать интерфейс одноцветным. Разные сущности должны иметь разные, но сдержанные акценты: люди, задачи, документы, направления.

## 16. Доступность

Минимальные требования:

- контраст текста и интерактивных элементов не ниже WCAG AA;
- все действия доступны с клавиатуры;
- видимый focus state у ссылок, кнопок, фильтров и карточек;
- иконки имеют `aria-label` или соседний текст;
- статусы не кодируются только цветом, рядом всегда есть текст: `Просрочено`, `На этой неделе`, `Нет описания события`;
- интерактивные элементы имеют размер не меньше 44px на touch-устройствах;
- поддерживается `prefers-reduced-motion`;
- таблицы на мобильном заменяются карточками, чтобы не требовать горизонтальной прокрутки;
- PDF/изображения имеют понятные текстовые ссылки, если preview не загрузился и доступен `publicUrl`.

## 17. Состояния интерфейса

Нужно явно предусмотреть:

- нет событий;
- нет документов;
- нет задач;
- документ не найден;
- событие без PDF;
- PDF без события;
- неизвестная дата;
- неизвестная клиника;
- неизвестный врач;

Для обычного зрителя технические ошибки парсинга не должны занимать главный экран. На главной можно показать только мягкое пользовательское предупреждение вроде “Есть необработанные документы”. Полный `DataQualityPanel` живёт в `/owner-guide` или отдельном служебном разделе с непервичной ссылкой.

## 18. Проверки качества данных

Нужна отдельная команда:

```powershell
npm run validate:data
```

Severity matrix:

| Проблема | Severity | Поведение |
|---|---|---|
| Невозможно прочитать хранилище | fatal | build падает |
| Raw PDF/JPG/PNG найден в production `dist` | fatal | build падает |
| Невозможно скопировать документы в приватной сборке | fatal | приватная сборка падает |
| Два события с одинаковым валидным `id` | fatal | build падает |
| Событие без `person` | skip | не строить публичный route, показать в служебном `DataQualityPanel` |
| Событие без `date` | skip | не строить публичный route, показать в служебном `DataQualityPanel` |
| Событие без `id` | warn | сгенерировать fallback slug по пути, показать предупреждение |
| `source_files` указывает на несуществующий файл | warn | событие остаётся, документ помечается как отсутствующий |
| PDF найден, но не связан ни с одним событием | warn | показать как “документ без описания события” |
| Дата не соответствует `YYYY-MM-DD` | warn/skip | warn, если можно распарсить; skip, если нельзя |
| Файл лежит в `04 Входящие` | info | показывать только в служебной зоне, если `publishInbox: true` |

Также нужны unit-тесты или фикстуры для:

- frontmatter parsing;
- извлечения секций markdown;
- парсинга профиля из тела заметки;
- `source_files` matching;
- slug generation;
- date parsing;
- duplicate detection;
- link checks;
- классификации документов без описания события.

## 19. GitHub Pages и deploy

Нужно добавить:

```text
.github/workflows/deploy-dashboard.yml
```

Workflow:

1. Checkout repo.
2. Setup Node.js.
3. `cd "06 Сайт"`.
4. `npm ci`.
5. `npm run validate:data`.
6. `npm run validate:assets`.
7. `npm run build`.
8. Deploy `06 Сайт/dist` to GitHub Pages.

Минимальный workflow должен включать:

```yaml
name: Deploy dashboard

on:
  push:
    branches: [main]
  workflow_dispatch:
  schedule:
    - cron: "15 2 * * *"

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: npm
          cache-dependency-path: "06 Сайт/package-lock.json"
      - uses: actions/configure-pages@v5
      - run: npm ci
        working-directory: "06 Сайт"
      - run: npm run validate:data
        working-directory: "06 Сайт"
      - run: npm run validate:assets
        working-directory: "06 Сайт"
      - run: npm run build
        working-directory: "06 Сайт"
      - uses: actions/upload-pages-artifact@v3
        with:
          path: "06 Сайт/dist"
  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - id: deployment
        uses: actions/deploy-pages@v4
```

Для GitHub Pages нужно учесть base path:

```text
/MedsDataBase/
```

Все ссылки и asset URL должны работать с этим base path.

Предусловие: репозиторий должен оставаться приватным. Если GitHub Pages не даст включить публикацию из private repo, запасной вариант — деплоить тот же публичный `dist` через Cloudflare Pages/Vercel или закрытый сервер, но не раскрывать исходный репозиторий с медицинскими документами.

## 23. Риски

### Риск: длинные кириллические пути

Меры:

- использовать внутренние `id`;
- не публиковать raw-документы в production `dist`;
- в приватной сборке копировать документы в `dist/files/documents` с hash;
- не строить URL напрямую из полного пути.

### Риск: часть PDF не имеет markdown-описания события

Меры:

- показывать такие файлы на странице документов;
- добавить мягкое предупреждение “Есть документы без описания события”;
- не скрывать данные только потому, что они ещё не нормализованы.

### Риск: пользователь думает, что кнопка “Добавить” меняет данные

Меры:

- не делать форму добавления;
- не делать первичную кнопку “Добавить запись”;
- служебную инструкцию держать в `/owner-guide`;
- в публичном UI использовать только действия просмотра.

### Риск: шаблоны попадут в дашборд как реальные события

Меры:

- парсить события только из `01 Члены семьи`;
- исключать `03 Шаблоны` всегда;
- постепенно перевести шаблоны на `type: template`;
- добавить тест, что `id: null` и `date: YYYY-MM-DD` из шаблонов не попадают в `DashboardData`.

### Риск: статический сайт устаревает по датам

Меры:

- статус задач и возраст вычислять на клиенте;
- хранить дату сборки только как справочную;
- добавить nightly rebuild, чтобы поисковый индекс и дата обновления регулярно освежались.

### Риск: build падает из-за одной плохой заметки

Меры:

- мягкая валидация;
- предупреждения в build log;
- служебный `DataQualityPanel` в `/owner-guide`;
- падать только при критичных нарушениях.

### Риск: публичность сайта

Меры:

- не добавлять секреты;
- не хранить токены;
- помнить, что всё в `dist` публично;
- не копировать PDF/JPG/PNG в production `dist`;
- держать `06 Сайт/src/generated/*.json` вне git;
- проверять Pages-артефакт командой `npm run validate:pages-artifact`;
- при смене требований к приватности перенести публикацию за Cloudflare Access или на сервер.

## 24. Критерии готовности MVP

MVP считается готовым, если:

- сайт собирается локально без ошибок;
- `npm run validate:data` проходит и показывает понятные предупреждения;
- сайт публикуется на GitHub Pages;
- главная показывает всех людей, последние события и ближайшие задачи;
- можно открыть карточку любого события;
- можно увидеть карточку связанного PDF/изображения без публикации оригинала;
- в приватной сборке с документами можно открыть связанный PDF/изображение;
- можно открыть карточку документа без потери контекста события;
- PDF без описания события видны на странице документов;
- фильтры и поиск работают без перезагрузки;
- сайт нормально выглядит на desktop и mobile;
- после `git push` обновления появляются на сайте автоматически;
- в основном интерфейсе нет сценариев, которые создают иллюзию записи данных;
- обычный зритель не видит GitHub/Obsidian/git/Codex как часть сценария просмотра.
