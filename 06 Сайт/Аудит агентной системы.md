# Аудит агентной системы

Дата: 2026-05-07
Область: Obsidian-хранилище, агенты обработки, производные JSON-слои, статический сайт, GitHub Pages workflow.

## Короткий вывод

Система уже выросла из дашборда в настоящую агентную операционку: есть intake, promote, metrics, tasks, watchlist, doctor summaries, валидатор и сайт. Базовая сборка проходит, данные связаны, документов без событий сейчас нет.

Но есть несколько существенных рисков:

1. **P0: приватность публикации.** Текущий GitHub Pages workflow публикует не только HTML, но и исходные PDF/JPG/PNG медицинские документы.
2. **P0/P1: данные в git и generated JSON являются leakage surface.** Даже без PDF, `dashboard-data.json` и `document-manifest.json` раскрывают имена, даты, структуру событий, пути и названия документов.
3. **P1: агенты могут тихо потерять данные.** Некоторые JSON read errors превращаются в пустые массивы и потом могут быть перезаписаны как "норма".
4. **P1: promote не атомарный.** При сбое между переносом исходника и записью события база может застрять в промежуточном состоянии.
5. **P1: operational UX неполный.** Нет единого экрана, где видно входящие, черновики, ошибки, кандидаты показателей, stale items и следующий шаг.

## Findings

### P0. GitHub Pages публикует исходные медицинские документы

Файлы:

- `.github/workflows/deploy-dashboard.yml`
- `06 Сайт/scripts/dashboard-lib.mjs`
- `06 Сайт/src/pages/documents/[documentSlug].astro`
- `06 Сайт/astro.config.mjs`

Проблема: workflow публикует `06 Сайт/dist` в GitHub Pages. `copyDocuments()` копирует документы в `dist/files/documents`, а страницы документов дают открыть оригинал. Это означает, что при включенном Pages наружу уходят не только карточки и события, но и PDF/JPG/PNG.

Последствие: полная утечка семейной медицинской базы при публичном Pages.

Mitigation:

- Временно выключить публикацию raw documents в production.
- Сделать `copyDocuments` opt-in только для local/private build.
- Добавить CI guard: production artifact не должен содержать `.pdf`, `.jpg`, `.jpeg`, `.png`.
- Для сайта оставить карточки документов без public raw URL или перейти на auth-gated hosting/signed URLs.

### P0. Репозиторий нельзя делать публичным

Файл:

- `06 Сайт/План реализации дашборда.md`

Проблема: в плане есть допущение публичности ради Pages. Для медицинской базы это нельзя считать безопасной моделью, потому что в git лежит история заметок и исходников.

Mitigation:

- Зафиксировать policy: **repo never public**.
- Хранить raw medical docs вне git либо в отдельном приватном encrypted storage.
- Если нужна публичная демо-версия, использовать только synthetic/redacted dataset.

### P1. Поврежденный JSON может стать пустым и перезаписаться

Файлы:

- `06 Сайт/scripts/metrics-agent.mjs`
- `06 Сайт/scripts/tasks-agent.mjs`
- `06 Сайт/scripts/dashboard-lib.mjs`

Проблема: `readJson()` в metrics/tasks и read-функции dashboard layer ловят parse/read errors и возвращают fallback/empty arrays. После этого agents могут записать файл обратно как будто данных нет.

Последствие: битый или частично записанный `metrics.json`/`tasks.json` может привести к тихой потере записей или к сайту с "пустой нормой".

Mitigation:

- Различать `ENOENT` и parse error.
- Parse error должен быть hard fail.
- Добавить `atomicWriteJson()` через temp file + rename.
- Перед записью делать schema validation и backup.
- Build должен показывать fatal data issue, если derived JSON не читается.

### P1. `intake-agent promote` не атомарный

Файл:

- `06 Сайт/scripts/intake-agent.mjs`

Проблема: promote копирует исходники и переносит их в processed до записи event-файла. Если запись события или перенос draft падает, approved draft остается, но `source_files` уже могут указывать на перемещенный inbox path.

Последствие: повторный запуск может падать на missing source, а оператор получит источник в `90 Обработано` без корректного события.

Mitigation:

- Сделать preflight всех путей до любых move/copy.
- Писать event во временный файл.
- Завершать операцию через transaction journal или staged rename.
- Добавить recovery report для незавершенных promote-операций.

### P1. `tasks-agent` стирает статус и ручные правки своих задач

Файл:

- `06 Сайт/scripts/tasks-agent.mjs`

Проблема: agent-generated tasks всегда создаются заново со `status: open`. Старые records с `source_agent: tasks-agent` не сохраняются. Если пользователь закрыл/отменил/переименовал generated task, следующий scan может воскресить ее как open.

Mitigation:

- Merge по `dedupe_key`.
- Сохранять user-owned поля: `status`, `reviewed_at`, `closed_at`, `notes`, manual priority override.
- Не удалять `done/cancelled/rejected` без явного archive policy.

### P1. `metrics-agent scan` может уничтожить незавершенный review state

Файл:

- `06 Сайт/scripts/metrics-agent.mjs`

Проблема: scan каждый раз заново пишет `metric_candidates.json` и `Проверка показателей.md`. Если пользователь уже поставил `approved` или галочки, но случайно запустил scan до apply, review state может потеряться.

Mitigation:

- Перед scan читать старые candidates и checked review ids.
- Переносить `approved/rejected/imported/reviewed_at` по `id` или `dedupe_key`.
- Добавить warning, если есть pending approved candidates.

### P1. `prebuild` не обновляет все derived layers

Файл:

- `06 Сайт/package.json`

Проблема: `prebuild` запускает validator, doctor summaries, data generation и assets validation, но не запускает metrics/tasks/watchlist. Сайт может успешно собраться со старыми показателями, задачами и watchlist.

Mitigation:

- Добавить `agent:all` или `refresh:derived`.
- Build должен проверять freshness через `generated_at` и source hash.
- Если слой устарел, build должен падать или показывать explicit stale warning.

### P1. Нет операционного экрана агентной очереди

Файлы:

- `06 Сайт/scripts/dashboard-lib.mjs`
- `06 Сайт/src/layouts/AppLayout.astro`
- `06 Сайт/src/pages/owner-guide.astro`

Проблема: сайт показывает итоговые данные, но не показывает состояние агентной системы: сколько новых файлов, сколько AI drafts, что на проверке, что approved, что в errors, когда последний запуск, какие candidates ждут apply.

Mitigation:

- Сделать страницу `/operations`.
- Показывать counts по `04 Входящие/*`, `intake-state.json`, `metric_candidates.json`, data issues, stale generated layers.
- Добавить "следующий шаг": `run intake`, `approve drafts`, `apply metrics`, `refresh tasks`, `publish`.

### P1. Human review заявлен, но не enforced везде

Файлы:

- `06 Сайт/scripts/intake-agent.mjs`
- `06 Сайт/scripts/metrics-agent.mjs`
- `06 Сайт/scripts/tasks-agent.mjs`
- `07 Показатели/metrics.json`
- `08 Задачи/tasks.json`

Проблема: в intake promote достаточно `status: approved` в frontmatter; чеклист и `needs_human_review` не enforced. Metrics можно импортировать по галочке или status без обязательного `reviewed_by/reviewed_at`. Tasks пишутся напрямую в `tasks.json`, без candidates review.

Mitigation:

- Для approved требовать `reviewed_by`, `reviewed_at`, `review_version`.
- В intake promote проверять checklist или явную machine-readable approval секцию.
- Tasks перевести на `task_candidates.json` + apply, как metrics.
- CI должен ругаться на approved derived medical facts без review metadata.

### P1. Validation gates слишком узкие

Файл:

- `06 Сайт/scripts/validate-operational-layer.mjs`

Проблема: валидатор проверяет директории, JSON parse, уникальность ids, базовые person/metric ids, но не проверяет полный контракт metrics/tasks/watchlist, `source_event_id`, `dedupe_key`, даты, статусы, required fields и существование всех source paths.

Mitigation:

- Ввести JSON Schema или Zod-подобный runtime validator для каждого output.
- Проверять source_event_id/source_event_path/source_files.
- Проверять допустимые статусы и обязательные поля.
- Отдельно валидировать medical_event provenance.

### P1. Generated data tracked в git

Файлы:

- `06 Сайт/src/generated/dashboard-data.json`
- `06 Сайт/src/generated/document-manifest.json`

Проблема: generated JSON содержит имена, даты, пути, public URLs и структуру базы. Он tracked в git и остается в истории.

Mitigation:

- Не коммитить `src/generated/*.json`.
- Генерировать в CI/private build.
- Добавить `.gitignore` и CI check.
- Для публичного сайта делать отдельную redacted projection.

### P2. `source_files` и provenance проверяются слишком мягко

Файлы:

- `02 Справочники/agent_contract.json`
- `06 Сайт/scripts/validate-operational-layer.mjs`
- отдельные `medical_event` заметки

Проблема: контракт требует `source_files`, но в базе есть события с пустыми `source_files`. Иногда это может быть ручная заметка, но тогда это должно быть явно отмечено.

Mitigation:

- `medical_event.source_files` обязателен, кроме `source_type: manual_note`.
- Для manual note требовать `provenance_note`, автора и дату.
- В UI маркировать "без первоисточника".

### P2. Факты, OCR и интерпретации смешаны в одной схеме

Файл:

- `06 Сайт/scripts/intake-agent.mjs`

Проблема: `summary` и `tasks` являются строками. В них смешиваются факты из документа, OCR-пересказ и предложения агента.

Mitigation:

- Разделить schema:
  - `facts[]` с source locator.
  - `uncertain_extractions[]`.
  - `agent_suggestions[]`.
  - `tasks_candidates[]`.
- Не переносить suggestions в события/задачи без review.

### P2. Doctor summaries без claim-level evidence

Файл:

- `06 Сайт/scripts/doctor-summary-agent.mjs`

Проблема: сводка содержит пункты, а источники вынесены отдельно. Нельзя сразу увидеть, из какого события/документа взят каждый тезис.

Mitigation:

- Хранить summary/track/nextActions как объекты `{ text, eventId, documentIds }`.
- На странице сводки рядом с каждым пунктом показывать ссылку на событие/документ.

### P2. Doctor summaries удаляют output до успешной генерации

Файл:

- `06 Сайт/scripts/doctor-summary-agent.mjs`

Проблема: agent удаляет `05 Индексы/Сводки врачу`, потом пишет новые файлы. При сбое получится пустая или частичная директория.

Mitigation:

- Генерировать во временную директорию.
- Проверять количество markdown и JSON records.
- Потом атомарно менять директорию.

### P2. Watchlist живет вне общего контракта

Файл:

- `06 Сайт/scripts/Update-Watchlist.ps1`

Проблема: люди и зоны хардкодятся в PowerShell, `created_at` пересоздается, `related_event_ids`/`related_task_ids` пустые.

Mitigation:

- Перенести zones/person references в JSON справочник.
- Сохранять stable `created_at` из предыдущего watchlist.
- Заполнять `related_event_ids`/`related_task_ids` или убрать поля из контракта.

### P2. Контакты третьих лиц могут публиковаться как medical_event

Файл:

- события/заметки с контактами специалистов

Проблема: контактная информация третьего лица может попасть в medical event, search и dashboard.

Mitigation:

- Ввести `type: private_contact` или `publish: false`.
- Redaction телефонов/email/ссылок для публичного/шаренного сайта.
- Исключить private contacts из search/dashboard.

### P3. `.env` лежит в рабочем дереве

Файлы:

- `.env`
- `.gitignore`
- `.env.example`

Проблема: `.env` не tracked и ignored, но находится в repo root. Это не git leak сейчас, но риск при архивах, скриншотах, бэкапах и ручных копиях.

Mitigation:

- Перенести ключ в user-level env/secret manager.
- Добавить pre-commit secret scan.
- Ротировать ключ после любых публичных демонстраций архива/экрана.

### P3. Нет locks/atomic writes в большинстве write paths

Файлы:

- `06 Сайт/scripts/intake-agent.mjs`
- `06 Сайт/scripts/metrics-agent.mjs`
- `06 Сайт/scripts/tasks-agent.mjs`
- `06 Сайт/scripts/dashboard-lib.mjs`

Проблема: прямые `writeFile` без temp+rename и lock. Конкурентный запуск или прерывание процесса даст last-writer-wins или частичный файл.

Mitigation:

- Общий helper:
  - `readStrictJson()`
  - `validateJsonSchema()`
  - `atomicWriteJson()`
  - `withFileLock()`

### P3. Watchlist скрывает темы без источников

Файл:

- `06 Сайт/src/pages/watchlist.astro`

Проблема: если тема задана, но источников нет, она может исчезнуть с экрана. Для operational watchlist это плохой режим отказа.

Mitigation:

- Показывать такие темы как "требует источника".
- Добавлять data issue вместо скрытия.

## Рекомендуемый порядок исправлений

### Блок 1. Privacy stop-loss

1. Отключить публикацию raw documents в Pages artifact.
2. Добавить CI guard на `.pdf/.jpg/.jpeg/.png` в `dist`.
3. Добавить policy `repo never public`.
4. Перестать коммитить generated JSON или сделать redacted projection.

### Блок 2. Защита от потери данных

1. `readStrictJson`: parse error = hard fail.
2. `atomicWriteJson` + backup.
3. Fix `tasks-agent` merge по `dedupe_key`.
4. Fix `metrics-agent scan` preserve review state.
5. Fix `doctor-summary-agent` temp output dir.

### Блок 3. Надежный promote

1. Preflight всех source/draft/event paths.
2. Transaction journal.
3. Event temp file -> copy docs -> move source/draft -> finalize state.
4. Recovery command/report.

### Блок 4. Enforced human review

1. `approved_by`, `approved_at`, `review_version`.
2. Intake approve command вместо ручного frontmatter.
3. Task candidates + apply.
4. Metrics approval metadata mandatory.

### Блок 5. Operations dashboard

1. `/operations` страница.
2. Counts: inbox folders, candidates, errors, stale generated layers.
3. Последний запуск агентов.
4. Next action hints.
5. Явные warnings/fatal issues на сайте.

### Блок 6. Traceability

1. Claim-level evidence для doctor summaries.
2. Facts vs interpretations schema в intake.
3. Provenance rules для manual notes.
4. Private contacts exclusion/redaction.

## Итоговая оценка

Текущая система функционально сильная: она уже связывает документы, события, задачи, показатели, наблюдение и сводки врачу. Основной риск теперь не в отсутствии возможностей, а в надежности и приватности operational layer.

Главная инженерная задача следующей итерации: **сделать систему fail-loud, privacy-safe и review-enforced**. После этого уже имеет смысл улучшать визуал и добавлять чат поверх базы.
