# Показатели

`metrics.json` — будущий структурный слой для анализов и измеримых показателей.

Он нужен для вопросов вроде:

- “Как менялся холестерин?”
- “Когда последний раз проверяли ТТГ?”
- “Какие показатели были вне референса?”

## Принцип

Markdown-событие остается основной медицинской заметкой. `metrics.json` хранит только атомарные значения показателей со ссылкой на источник.

## Референсы

Референс показателя берётся из конкретного лабораторного документа, а не из общего справочника норм.

Почему так:

- нормы зависят от лаборатории, метода, единиц, пола, возраста, беременности и клинического контекста;
- один и тот же canonical-показатель (`tsh`, `ferritin`, `ldl_cholesterol`) может иметь разные референсы в разных бланках;
- если в документе нет референса или он не был извлечён, показатель должен отображаться как “Нет референса”, без тревожной окраски.

Минимальная модель записи:

- `value`, `numeric_value`, `unit` — фактическое значение и единицы из документа;
- `reference_low`, `reference_high`, `reference_text` — исходный референс из документа;
- `source_files`, `source_event_id`, `source_text` — откуда взят показатель;
- `metric_id` — canonical id из справочника, чтобы связывать разные названия одного показателя.

Правило оценки:

- если есть лабораторный флаг отклонения, его можно учитывать;
- если есть `reference_low` / `reference_high`, статус можно вычислять относительно этих границ;
- если есть только односторонний референс (`< 3,00`, `> 10`), он хранится как верхняя или нижняя граница;
- если референса нет, статус остаётся неизвестным.

Следующая цель для агента анализов: при извлечении из PDF/PNG сохранять не только значение, но и лабораторию, метод/материал при наличии, единицы и исходный референс из той же строки или таблицы.

## Проверка кандидатов

Новые значения сначала попадают в `metric_candidates.json` и в человекочитаемый файл `Проверка показателей.md`.

Что проверять перед одобрением:

- человек и дата;
- canonical показатель `metric_id` и видимое название;
- значение, единицы и референс;
- `source_event_path` или `source_draft_path`;
- `source_text`, `source_window`, `source_evidence`;
- `value_confidence` и `reference_confidence`;
- `review_warning`, если агент пометил неоднозначное совпадение.

Разрешённые ручные изменения в `metric_candidates.json`:

- поставить `status: "approved"`, если строка полностью совпадает с источником;
- поставить `status: "rejected"`, если строка неверная;
- добавить `review_comment`, если нужно объяснить решение.

Не нужно вручную переносить строки в `metrics.json`. После проверки запускать из `06 Сайт`:

```powershell
npm run agent:metrics -- apply
```

Если сомневаешься в значении, единицах, референсе или canonical id, кандидат остаётся `needs_review`.
