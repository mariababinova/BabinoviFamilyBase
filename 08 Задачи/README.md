# Задачи

`tasks.json` — будущий явный слой контрольных задач.

Сейчас дашборд уже умеет вычислять задачи из `follow_up_date` в медицинских событиях. Этот файл нужен для следующего шага: чтобы агент мог создавать отдельные задачи, связанные с документами, событиями и зонами наблюдения.

## Примеры задач

- контроль липидограммы;
- повторить УЗИ;
- записаться к врачу;
- взять документ на прием;
- проверить, что рекомендация врача выполнена.

## Агент задач

Команда запускается из `06 Сайт`:

```powershell
npm run agent:tasks
npm run agent:tasks -- scan
npm run agent:tasks -- apply
```

Агент читает проверенные медицинские события из `01 Члены семьи`, ищет контрольные формулировки в разделах `Что делать дальше`, `Назначения`, `Что важно отследить`, `Контроль в июле`, `Планово` и обновляет `task_candidates.json` / `tasks.json`.

Записи с `source_agent: "tasks-agent"` пересоздаются при каждом запуске. Ручные записи без этого поля сохраняются.

Режимы:

- `scan` — обновляет кандидаты, но не переносит новые рекомендации в подтверждённые задачи;
- `apply` — переносит только кандидаты со статусом `approved`;
- обычный `npm run agent:tasks` — совместимый режим `refresh`: сканирует и применяет уже одобренные кандидаты.

## Проверка кандидатов

Новые рекомендации должны оставаться в `task_candidates.json`, пока человек их не подтвердит.

Что проверять:

- `person`, `due_date`, `title`, `action_text`;
- `source_event_path`, `source_files`, `source_text`;
- `reason` — почему агент решил, что это задача;
- `extraction_confidence`;
- нет ли это назначения лекарства, справочной фразы или задачи без реального действия.

Разрешённые ручные изменения:

- `status`, `candidate_status` или `review_status`: `approved` / `rejected` / `needs_review`;
- `review_comment`;
- пользовательские поля вроде `priority_override` или `title_override`, если задача верная, но текст нужно поправить.

После проверки запускать:

```powershell
npm run agent:tasks -- apply
```
